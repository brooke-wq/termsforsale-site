5. File-Based Dedup System

How It Works
  • Location: /root/termsforsale-site/jobs/sent-log.json on Droplet
  • Module: jobs/sent-log.js
  • Key format: {contactId}-{dealId}-{step} (e.g. abc123-def456-d0)
  • Works only on Droplet — Netlify functions are serverless/ephemeral, so file dedup is skipped there
  • notify-buyers.js: checks before sending, marks after (per contact per deal)
  • deal-follow-up.js: checks before each Day 0/1/2, marks after. 1 msg/contact/run cap.
  • ceo-briefing.js: daily dedup key — won't double-SMS on re-runs
  • weekly-synthesis.js: weekly dedup key — won't double-SMS on re-runs
  • follow-up-nudge.js: per-contact dedup — won't re-nudge same contact
  • Tag-based dedup in GHL still works as a secondary layer

7. Authentication System

How Auth Works
  • Signup → password hashed (PBKDF2, 10,000 iterations + random salt) and stored on GHL contact as tfs_password_hash custom field
  • Login → email lookup in GHL, password verified against stored hash. Wrong password = rejected with error message.
  • Legacy users (accounts created before April 3, 2026) have no stored hash — they can still log in but are flagged as legacyUser. Once they reset their password, they get a proper hash.
  • Password reset → 6-digit code sent via email + SMS, expires in 15 minutes. Code stored on contact as tfs_reset_code. After verification, new password hash replaces old one.
Auth Endpoints
EndpointPurpose
/api/auth-signupCreate account — hashes password, creates GHL contact, sends welcome SMS + email, redirects to buying criteria
/api/auth-loginLogin — verifies password hash, returns user data with hasBuyBox and isVip flags from GHL tags
/api/auth-resetPassword reset — Step 1: send 6-digit code. Step 2: verify code + set new password.
GHL Custom Fields (Auth)
Field KeyPurpose
tfs_password_hashPBKDF2 hash of user's password (salt:hash format). Set on signup and password reset.
tfs_reset_code6-digit reset code + expiry timestamp (code:expiry format). Cleared after use.
IMPORTANT: The tfs_password_hash and tfs_reset_code custom fields must exist in GHL for auth to work. If they don't exist, create them in GHL → Settings → Custom Fields → Contact Fields as "Large Text" type.

8. Deal Status Flow

New Submission → Pending Items → Ready for Review → Accepted → Actively Marketing → Assignment Sent → Assigned with EMD → Closed

  • "Actively Marketing" = deal shows on website + triggers buyer notifications within 30 min
  • "Closed" + Date Funded = shows in CEO briefing + revenue reports
  • Changing status to anything other than "Actively Marketing" removes it from the website

9. VA Responsibilities (Manual Processes)

Daily Tasks
  • Enter all new deal data in Notion with ALL required fields
  • Upload property photos to Google Drive folder, link URL in Notion "Photos" field
  • Photos sort alphabetically by filename — name the front-of-house photo first (e.g. 01-front.jpg)
  • NEVER include street addresses in outbound marketing (emails, SMS, deal alerts, blog posts)
  • Set deal status to "Actively Marketing" when ready to list
  • Review CEO briefing notes in GHL for action items
  • Monitor GHL for flagged leads (flag-escalate tag)
When a Deal Closes
  • Set Date Funded in Notion (the date funds were received)
  • Set Amount Funded in Notion (total revenue from this deal)
  • Change Deal Status to "Closed"
  • These 3 fields trigger the revenue tracker and CEO briefing automatically
Blog Posts (Deal Spotlights)
  • Use the VA Post Builder at /api/create-post to create deal spotlight blog posts
  • Posts are added to /blog/posts-index.json and appear on the blog page automatically

13. Domain Switch Checklist

  2. In DNS provider: Update A record @ → 75.2.60.5
  3. In DNS provider: Update CNAME www → jolly-marzipan-8fc443.netlify.app
